# Notes: foreach vs forall (Professional)

*From `3. foreach vs forall.txt`.*

---

## 1. Deferred execution still applies

Like sequential LINQ, PLINQ queries are **deferred**: query operator code runs when the query is enumerated (e.g. `foreach`, `Count`, `ToList`, `ForAll`), not when the query is constructed.

---

## 2. foreach implies merging to a single consumer

When enumerated with `foreach`, consumption occurs on the enumerator thread (often the main thread). Worker partitions produce results that must be **merged** into a single stream, which is why merge buffering (`WithMergeOptions`) is relevant.

---

## 3. ForAll avoids the single-consumer merge

`ForAll(Action<T>)` executes the consumer action in parallel. The producer and consumer can run on the same worker threads, reducing merge overhead and eliminating the “main thread consumption” pattern.

---

## 4. Example shape

```csharp
var items = Enumerable.Range(1, 200);

var evenNumbers = items.AsParallel().Where(x =>
{
    Console.WriteLine($"Processing number {x}; Thread Id: {Thread.CurrentThread.ManagedThreadId}");
    return (x % 2 == 0);
});

evenNumbers.ForAll(item =>
{
    Console.WriteLine($"{item}: Thread Id: {Thread.CurrentThread.ManagedThreadId}");
});
```

---

## Summary table

| Consumption | Characteristics |
|------------|------------------|
| `foreach` | Single consumer; merge/buffering required. |
| `ForAll` | Parallel consumer; less/no merge step. |
