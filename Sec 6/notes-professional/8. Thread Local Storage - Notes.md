# Notes: Thread Local Storage (Professional)

*From `8. Thread Local Storage.txt`.*

---

## 1. Motivation: reduce contention on shared state

A shared accumulator (e.g. `sum`) protected by a lock in every iteration can become a major bottleneck for large ranges: workers spend significant time blocked on the lock rather than doing useful work.

---

## 2. Parallel.For thread-local overload

The TLS overload provides:

- `localInit`: initializes a local value per worker
- `body`: updates and returns the local value per iteration
- `localFinally`: aggregates each worker's local value into a shared result (invoked once per worker), requiring minimal synchronization

---

## 3. Example code

```csharp
int[] array = Enumerable.Range(1, 10).ToArray();

int sum = 0;
object lockSum = new object();

Parallel.For(
    0,
    array.Length,
    () => 0,
    (i, state, tls) =>
    {
        tls += array[i];
        return tls;
    },
    tls =>
    {
        lock (lockSum)
        {
            sum += tls;
            Console.WriteLine($"The task id: {Task.CurrentId}");
        }
    }
);

Console.WriteLine($"The sum is {sum}");
Console.ReadLine();
```

---

## Summary table

| Mechanism | Effect |
|----------|--------|
| Lock per iteration | High contention; poor scalability. |
| TLS (`localInit/body/localFinally`) | Work is local; synchronization only during aggregation. |
