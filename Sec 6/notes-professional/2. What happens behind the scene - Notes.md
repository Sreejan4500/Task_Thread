*From `2. What happens behind the scene.txt`.*

---

## 1. Data partitioning

`Parallel.For` partitions the iteration space into **chunks** and schedules those chunks onto a limited set of workers. This is why thread/task IDs observed during execution are far fewer than the number of iterations.

---

## 2. Worker selection (mostly thread pool)

Parallel loops typically execute on **thread-pool** threads. The runtime may occasionally use a non-pool thread depending on scheduling decisions, but the model is “runtime-managed workers” rather than developer-created threads.

---

## 3. Runtime heuristics

The implementation attempts to choose a reasonable degree of parallelism (often aligned with logical CPU cores, but influenced by additional factors). If you need full control over partitioning and worker management, you drop down to explicit threads/tasks.

---

## 4. Blocking semantics

`Parallel.For`/`ForEach`/`Invoke` are **blocking** calls: code after the call executes only after all scheduled work completes (or terminates due to stop/break/exception/cancellation).

---

## Summary table

| Topic | Key point |
|------|-----------|
| Partitioning | Iterations are chunked; not 1 worker per iteration. |
| Thread pool | Workers are usually pool threads. |

| Blocking | Caller continues only after loop completion. |
