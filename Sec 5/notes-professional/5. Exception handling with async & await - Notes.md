# Notes: Exception handling with async & await (Professional)

*From `5. Exception handling with async & await.txt`.*

---

## 1. Task exceptions vs await exception propagation

Without `await`, exceptions can remain **captured** on the task, and code often inspects them via `AggregateException` and `InnerExceptions`.

When using **await**, the awaiter **throws** an exception outward (propagates failure), so the application does not silently continue in a bad state.

---

## 2. await throws the “first” exception (from a multi-fault Task)

For `Task.WhenAll(...)`, multiple tasks can fault. Awaiting the combined task throws **one** exception (effectively the first observed), which can vary due to scheduling. The key behavior difference highlighted by the video:

- `Wait()`/`.Result` → typically `AggregateException`
- `await` → throws an inner exception directly

---

## 3. Example shape (conceptual)

```csharp
try
{
    var tasks = new[]
    {
        Task.Run(() => throw new InvalidOperationException("Invalid operation exception")),
        Task.Run(() => throw new ArgumentNullException("Argument null exception")),
        Task.Run(() => throw new Exception("General exception")),
    };

    await Task.WhenAll(tasks); // throws one exception
}
catch (Exception ex)
{
    Console.WriteLine(ex.GetType().Name);
}
```

---

## Summary table

| Topic | Key point |
|------|-----------|
| Exceptions stored in tasks | Failures are captured until observed. |
| `Wait()` / `.Result` | Surfaces `AggregateException`. |
| `await` | Propagates an inner exception (one of the failures). |
| Ordering | Which failure is “first” is not deterministic under parallelism. |
