# Notes: Overview of Async & Await (Professional)

*From `1. Overview of Async & Await.txt`.*

---

## 1. async/await: sequential-style code without blocking

**async** and **await** let you write asynchronous code in a linear, “do this then that” style. At an **await**, the calling thread is released instead of blocking; the remainder of the method runs as a **continuation** when the awaited task completes.

---

## 2. Continuation semantics

Everything after an **await** is the **continuation**. It is scheduled to run when the awaited task completes (on a thread-pool thread or, when a synchronization context exists, on that context—e.g. UI thread).

---

## 3. await unwraps Task&lt;T&gt; to T

**await** on a **Task&lt;T&gt;** yields a value of type **T** directly. No need to use **.Result**; the compiler generates the continuation and result handling.

---

## 4. Exception behavior

**await** rethrows the **first** exception from the task’s **AggregateException** (if any). So you catch the actual exception type (e.g. **HttpRequestException**), not **AggregateException**, unlike **Wait()** / **Result**.

---

## 5. Synchronization context

**await** respects **SynchronizationContext** (and **TaskScheduler**). In UI apps, continuations typically run on the UI thread, so you don’t need manual **Invoke** for UI updates after **await**.

---

## Summary table

| Concept | Role |
|--------|------|
| **async** | Enables **await** in the method. |
| **await** | Yields; continuation runs when task completes. |
| **Continuation** | Code after **await**; runs on context/scheduler. |
| **Task&lt;T&gt; → T** | **await** returns **T** directly. |
| **Exceptions** | First inner exception is thrown by **await**. |
| **SynchronizationContext** | Continuations marshalled back (e.g. to UI thread). |
