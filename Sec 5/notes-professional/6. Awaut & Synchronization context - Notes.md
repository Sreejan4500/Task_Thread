# Notes: Awaut & Synchronization context (Professional)

*From `6. Awaut & Synchronization context.txt`.*

---

## 1. The problem: thread affinity in UI frameworks

UI frameworks (Windows Forms) require UI components to be updated only from the UI thread. Work done on other threads requires marshaling back (e.g. `Control.Invoke`) to avoid exceptions.

---

## 2. The behavior highlighted: await resumes on the captured context

The video’s key point: even though code after an **await** is a continuation, **await** captures the current **SynchronizationContext** and uses it to resume the continuation. This is why code after `await` in a UI event handler can update UI controls without explicit `Invoke`.

---

## 3. Example (Windows Forms shape)

```csharp
private async void button1_Click(object sender, EventArgs e)
{
    await ShowMessage("First Message", 3000);
}

private async Task ShowMessage(string message, int delay)
{
    await Task.Delay(delay);
    lblMessage.Text = message;
}
```

## Summary table

| Topic | Key point |
|------|-----------|
| Thread affinity | UI updates must occur on UI thread. |
| `Invoke` approach | Manual marshaling back to UI thread. |
| `await` approach | Captures context; continuation runs on captured context by default. |
